# Champion App Spec

## 1. 基本情報
- 企画名: Gimme
- 一言コンセプト: 家族の「もらい損ね」を見つけ、申請準備をクエスト化する世帯奪還アプリ
- Summit mode: thursday_original
- Build slot: Thursday Codex direct
- Idea frame: 2 original-future-demand
- Topic genre No: 2
- Topic genre: AI生活補助
- Shinchan選定番号: 1
- Top5内の元順位: 1位
- 想定ユーザー: 30〜50代の世帯主、家計管理者、子育て/住宅/介護/医療費などで制度活用したい家族
- 主な利用シーン: 家族情報を入れて、今月申請できる可能性がある補助金・控除・返金・解約忘れを確認する
- このアプリが最初に狙う価値: 「今月いくら取り戻せそうか」がすぐ見えること
- 正式リリース形態: Android/iOSネイティブモバイルアプリ
- Flutter Webの位置づけ: スマホ遠隔確認用の初期プレビュー

## 2. 収益性
- 収益モデル: 月額の世帯監視プラン + 成果報酬 + 優先申請/詳細分析プレミアム
- 継続収益モデル候補: 世帯監視、家族追加、制度更新アラート、期限通知、成果レポート
- 支払い主体: 世帯のお金を管理する人
- 支払い理由: 月額以上のお金が戻る可能性があるため
- 想定単価/LTV仮説: 個人月額980円、世帯プラン1,480〜1,980円、成果報酬5〜15%
- 解約されにくくする理由: 期限、制度変更、家族の年齢変化、控除条件が継続的に変わる
- 継続的に支払われる理由: 放置すると損をする恐怖と、実際に回収できた達成感
- 継続利用/再訪のきっかけ: 新しい奪還チャンス、期限接近、成果達成、家族イベント
- 課金ポイント: 世帯監視、優先アラート、申請チェックリスト、家族共有、成果レポート
- 無料範囲: モック診断、上位3件の奪還候補、簡易クエスト
- 有料化しても納得される理由: 金額で価値が見える
- 初期リリースで検証する収益仮説: 「取り戻せる見込み金額」が月額支払い意思を作るか
- 競合より支払う理由: 単なる家計簿ではなく、具体的な回収行動まで落とす
- ネイティブアプリ内課金/サブスクへ移す時の課金導線: 成果ダッシュボードと期限通知から世帯監視プランへ誘導

## 3. 元の議題との対応
- 元の議題: AI生活補助
- どの部分に直接応えるか: 家計・制度・手続きの見落としをAIが生活補助として発見する
- 議題から拡張した価値: 個人ではなく世帯単位で、支払い理由が明確な生活AIへ拡張

## 4. 解決する課題
- ユーザーが困っていること: 補助金や控除は複雑で、気づいた時には期限切れになりやすい
- 既存手段の弱いところ: 自治体サイト、家計簿、検索だけでは「自分が対象か」がわかりにくい
- このアプリでどう変わるか: 家族構成と生活イベントから、今やるべき申請準備がクエストとして見える

## 5. コア価値
- このアプリの核: 今月取り戻せる可能性がある金額と行動を一目で示す
- 絶対に失ってはいけない体験: 「自分の家庭にも関係ありそう」と感じる具体性
- ほかの案との差別化: 便利ではなく、金銭回収という強い支払い理由がある

## 6. Round 1 起点の原案価値
- 元になった案: Gimme
- provider: Gemini
- Round 1時点の価値: 個人の補助金、控除、解約忘れを全自動で検知・奪還するエージェント

## 7. Round 2 で強化された選定理由
- 選定理由: 「お金が戻る」は全LLMが最も強い支払い理由と判断
- 改善観点: 世帯単位、成果報酬、家族プラン、期限通知を明確化
- 他案より優先された理由: 収益性、説明の速さ、支払い主体の明確さが最も強い

## 8. Round 3 神追加案の採用内容
- 採用した追加案: 世帯奪還マトリクス、損失ウィジェット、奪還クエスト
- 取り込む理由: 家族全体の損失と回収額を見せることで単価と低解約性が上がる
- 取り込まない追加案と理由: 実際の自動申請代行は初期版では行わない。責任範囲が重いため、申請準備とモック追跡に留める

## 9. Round 4 最終採用判断
- Champion: Gimme

- Top5 finalist packetでの位置: 1位
- Runner-up: AI Personal Gatekeeper
- 最終判断: 今回はGimmeを制作し、NoManは保存企画とする
- 採点根拠: 金銭的便益、世帯課金、成果報酬、支払い主体の明確さ
- post-selection安全/法務/権利/実現性精査: 初期版は実制度への断定や自動申請を避け、モック制度データと申請準備体験にする

## 10. デザイン方針
- 画面密度: モバイル家計/フィンテック系として、情報量は多いが整理された密度
- トーン: 信頼できる、前向き、焦らせすぎない
- モバイル375px基準での優先順位: 今月の回収見込み、期限、次の一手を最上部に出す
- Android/iOSネイティブ画面としての優先順位: 通知、ウィジェット、家族共有、申請進捗
- Flutter Web確認版で崩してはいけない範囲: 主要カード、タブ、クエスト詳細、家族切替
- UI/見た目/文言をAI生成っぽく見せないための禁止事項: 汎用AIチャット風、薄い診断、抽象的な助言、説明だらけの画面
- AI活用機能がある場合、それを自然なアプリ体験として見せる条件: AIは裏側の照合役として見せ、画面は奪還候補と行動に集中する
- 通常の人気アプリと遜色ない品質にするための具体条件: 明確な階層、安定した余白、実在感のある金額/期限/ステータス

## 11. MVP定義
- 初回完成版で必須の機能: ダッシュボード、家族プロファイル、奪還候補一覧、クエスト詳細、ステップ完了、成果ダッシュボード、課金仮説表示
- 今回は後回しでよい機能: 実API連携、認証、実申請、銀行/マイナ連携、決済
- 完成品として最低限成立する条件: 家族情報に応じて候補が変わり、申請準備フローが最後まで動く
- 最高完成度を目指すための追加条件: モバイル幅での完成度、滑らかな状態更新、ローカル保存、説得力あるサンプル制度
- Flutter Web確認版で検証する範囲: 家族登録モック、奪還候補、申請クエスト、モック成果還元
- Android/iOS正式リリースまでに必要な追加範囲: 本人同意、制度DB、通知、課金、家族招待、セキュリティ

## 12. 画面設計
- 画面一覧: Dashboard、Opportunities、Quest Detail、Household、Insights、Plan
- 各画面の役割: Dashboardは全体金額、Opportunitiesは候補比較、Quest Detailは行動、Householdは家族条件、Insightsは成果、Planは収益仮説
- 初回表示画面: Dashboard
- 主な遷移: Dashboard -> Opportunity -> Quest Detail -> Step complete -> Results
- 空状態: 条件不足時は家族情報追加を促す
- ローディング状態: スキャン中カードを表示
- エラー状態: モック診断できない条件を明示して再入力へ戻す

## 13. ユーザーフロー
- ユーザーが最初にすること: 世帯情報を確認する
- 主操作の流れ: 候補を見る、クエストを開く、必要書類と手順を確認し、ステップを完了する
- 結果の確認方法: 今月の回収見込みと完了済み金額を見る
- 繰り返し利用時の流れ: 期限接近と新規候補を確認する
- ネイティブアプリでの再訪導線: プッシュ通知、ホームウィジェット、期限バッジ
- 通知/リマインド/ホーム画面起動を使う場合の方針: 初期Webでは通知をモック表示、正式版で実装

## 14. データ設計
- 入力する情報: 居住地、家族人数、子どもの年齢、住宅、医療費、サブスク、介護有無
- 保存する情報: 家族条件、候補ステータス、完了ステップ、成果金額
- 出力結果: 奪還候補、見込み金額、期限、必要書類、次の一手
- モックでよい部分: 制度DB、判定、申請ステータス、成果報酬
- ダミーデータで仮置きしてよい部分: 自治体名、制度名、金額、期限
- 将来API/DB化する部分: 公的制度DB、家計/税/医療/サブスク連携

## 15. 実装前提
- 開発形態: Android/iOSネイティブ正式リリースを前提にしたFlutterアプリ
- 初期デプロイ: Flutter Webの公開HTTPSプレビュー。これは確認用途であり、正式リリースではない
- 正式リリース方針: Android/iOSネイティブアプリとしてApp Store / Google Play公開を目指す
- ネイティブ移行方針: web-only前提を避け、画面、状態管理、課金、通知、永続化をネイティブ化しやすい構成にする
- 認証: 初回完成版では不要
- API / DB: 初回完成版ではモック
- 永続化: ローカル永続化
- 実装場所: `C:\dev\gimme`
- 公開先: オンラインの公開HTTPS URL。ローカルのみは確認完了扱いにしない。公開HTTPS URLは確認用で、正式リリース先はAndroid/iOS

## 16. 技術スタック前提
- フレームワーク: Flutter stable
- 言語: Dart
- UI: Material 3
- 状態管理: Flutter標準のStatefulWidget + ValueNotifier相当
- 永続化: shared_preferences
- API / DB: 初回完成版ではモック
- テスト: flutter test
- ビルド: flutter build web
- デプロイ: Flutter Web確認版として公開HTTPS URLへデプロイ
- デプロイ優先順位: Cloudflare Pages優先。不可ならVercel / Netlify / GitHub Pages / Firebase Hosting / 同等の静的ホスト
- 正式リリース準備: Android/iOSパッケージ化、アプリ内課金、通知、ストア審査対応は後続タスク
- リポジトリ: 必要に応じてGitHub

## 17. 実装優先順位
- 最優先: Dashboard、候補リスト、クエスト詳細、状態保存
- 次点: 世帯条件編集、成果/プラン表示、モバイルQA
- 今回は見送るもの: 実制度連携、決済、個人情報送信、実申請代行

## 18. 採用した要素 / 不採用にした要素
- 採用した要素: 世帯奪還、奪還クエスト、成果ダッシュボード、期限通知モック
- 不採用にした要素: 実API、自動申請、所得データ連携
- 不採用理由: 初期版では法務/責任/個人情報の扱いが重いため

## 19. リスクと仮置き方針
- 最大リスク: 制度判定の誤り
- 法務/権利リスク: 公的制度情報の更新、申請代行と助言責任
- 安全性/規制リスク: 金融/税務/行政手続きの断定表現
- 技術/制作コストリスク: 制度DBと家族情報連携
- リスク緩和策: 初期版はサンプル制度と申請準備チェックに限定し、断定せず確認を促す
- 不足情報がある場合の合理的デフォルト: 東京都在住の4人家族サンプル
- 止まらず進めるための仮置き条件: 金額と制度はデモデータ
- モックにしてよい条件: 判定、通知、成果、申請状況
- モックでは済ませない核心機能: 候補が見える、行動に落ちる、完了状態が残る
